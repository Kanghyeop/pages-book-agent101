## 문제: 정보는 많은데 글은 안 써진다

책을 쓰면서 겪은 가장 큰 어려움:

1. 정보량이 산만해진다

생각이 너무 많다. 이것도 넣고 싶고, 저것도 중요하고. 머릿속엔 수십 개 주제가 동시에 떠오른다. 어디서부터 시작해야 할지 모르겠고, 어디까지 써야 끝날지도 모르겠다.

2. 종결성 있는 글로 만들기 어렵다

정보는 모았는데, 이걸 "하나의 완결된 글"로 만드는 게 막막하다. 문단 순서, 논리 흐름, 결론까지 가는 길. 직접 쓰려고 하면 중간에 막히고, 수정하다 보면 원래 하고 싶던 말을 잃어버린다.

결론: 내가 직접 타이핑으로 쓰면 안 된다.

---

## 해결: 일부러 러프하게, 말로만 넣는다

핵심 원칙: 초안 단계에서는 텍스트를 직접 수정하지 않는다.

### 워크플로우

1단계: Letterly에 음성 메모

- 생각나는 대로 말한다 (5분~15분)
- 완벽한 문장 아니어도 됨
- "이거 중요한데", "이건 나중에", "예를 들면" 같은 메모성 발언 포함
- Letterly가 자동으로 텍스트 변환

2단계: Claude에게 명령

- Letterly 메모를 Claude에게 전달
- "이거 draft로 정리해줘"
- Claude가 구조화 + 문단 정리 + 톤 보정

3단계: Draft 폴더에 저장

- 파일명: `주제_MMDD-HHMM.md`
- 내용: Claude가 정리한 초안 (완벽하지 않아도 OK)
- CHAPTERS.md에 체크박스 추가

4단계: 나중에 다시 읽고 판단

- 며칠 뒤 다시 읽으면 객관적으로 보임
- "이건 쓸만하네" → final/ 이동
- "이건 아니다" → draft/archive/ 이동

---

## 왜 이렇게 하나?

1. 말이 글보다 빠르다

타이핑으로 5,000자 쓰려면 1시간. 말로 하면 15분.

2. 생각의 흐름이 끊기지 않는다

키보드로 치다 보면 "이 문장 어떻게 쓰지?" 고민하며 흐름이 끊긴다. 말은 그냥 흐른다.

3. 완벽주의를 차단한다

"이 문장 좀 이상한데..."라며 고치기 시작하면 30분이 날아간다. 음성 메모는 이미 끝났으니 고칠 수가 없다. 그게 좋다.

4. Claude가 구조를 잡아준다

내 머릿속 날것의 생각 → Claude가 문단 나누고, 제목 달고, 흐름 정리. 나는 방향만 제시하면 된다.

## 실제 예시

음성 메모 (러프):
```
"에이전트 시스템 만들 때 LangGraph 쓰면 좋은데,
근데 이게 뭐냐면 노드랑 엣지로 연결하는 거야.
상태가 흐르면서 각 노드에서 작업 수행하고.
예를 들어 근조화환 주문받으면, 전화 접수하고,
결제 확인하고, 발주하고, 배송 확인하고.
이런 흐름을 그래프로 만드는 거지.
조건부 분기도 가능하고."
```

Claude 정리 (Draft):
```markdown
# 근조화환 LangGraph 쉽게 이해하기

## LangGraph란?

여러 에이전트를 연결해서 흐름을 만드는 도구입니다.

### 핵심 개념
- State: 정보 꾸러미
- Node: 각 작업 단계
- Edge: 단계 간 연결

### 근조화환 예시
1. 전화 주문 접수
2. 결제 확인
3. 발주
4. 배송 확인
...
```

## 주의사항

이 워크플로우가 안 맞는 경우:

- 매우 정교한 논리 전개가 필요한 글 (논문, 법률 문서)
- 한 문장 한 문장이 치명적으로 중요한 글
- 이미 완성도 높은 초안이 있고 다듬기만 하면 되는 경우

이 워크플로우가 딱인 경우:

- 생각은 많은데 글이 안 써질 때
- 정보를 일단 쏟아내고 나중에 정리하고 싶을 때
- 완벽주의 때문에 첫 문장도 못 쓸 때
- 시간은 없는데 콘텐츠는 만들어야 할 때

---

## 결론

러프 드래프트의 가치

완벽한 초안을 만들려고 3시간 고민하는 것보다,
러프한 초안을 15분 만에 만들고 나중에 다듬는 게 낫다.

일부러 손 안 대기

초안 단계에서 텍스트 직접 수정하지 않기.
이게 핵심이다.

말 → Claude → Draft → (시간 지남) → 판단 → 정리

이 흐름을 지키면, 정보의 홍수에서 살아남을 수 있다.
